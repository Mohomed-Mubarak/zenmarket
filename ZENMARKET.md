## 1. Quick Start

### Requirements

- A local HTTP server (ES Modules do **not** work via `file://`)
- A modern browser (Chrome, Edge, Firefox, Safari)

### Option A — VS Code + Live Server (Recommended)

```bash
# 1. Unzip the project
unzip zenmarket-v29.zip
cd zenmarket-v29

# 2. Open in VS Code
code .

# 3. Install "Live Server" extension by Ritwick Dey
# 4. Right-click index.html → "Open with Live Server"
#    → Opens at http://127.0.0.1:5500
```

### Option B — Node.js

```bash
npx serve .
# Opens at http://localhost:3000
```

### Option C — Python

```bash
python3 -m http.server 8000
# Opens at http://localhost:8000
```

> ES Modules require a local server. Opening `index.html` via `file://` will fail with CORS errors.

### Default Admin Login

| Field    | Value                    |
|----------|--------------------------|
| URL      | `/admin/index.html`      |
| Email    | `admin@zenmarket.lk`     |
| Password | `admin123`               |

> **Change this password immediately** — Admin → Settings → Change Admin Password.

## 6. User Accounts & Authentication

### Registration Flow

When a customer creates an account at `/login.html`:

1. `register()` in `auth.js` validates email uniqueness
2. A user object is created with a unique `id` (`USR-{timestamp}`), `role: 'customer'`, and `createdAt`
3. Password is hashed using PBKDF2 (310,000 iterations) via the Web Crypto API
4. The user is saved to **both** localStorage stores:
   - `zm_registered_users` — keeps the password hash for login
   - `zm_users` — the shared store read by the admin panel (password stripped)
5. A session is set immediately — the user is logged in straight away
6. The account appears instantly in Admin → Users

### Login Flow

`login()` checks in this order:
1. `zm_registered_users` — self-registered accounts (PBKDF2 hash verified)
2. `zm_users` demo users — any password accepted in demo mode

### Session Management

| Function | Description |
|----------|-------------|
| `getSession()` | Returns current customer session or `null` |
| `setSession(user)` | Saves user to `zm_session` with a fresh UUID token |
| `clearSession()` | Removes `zm_session` |
| `isLoggedIn()` | Returns `true` if session exists |
| `getCurrentUser()` | Alias for `getSession()` |
| `logout()` | Clears session and redirects to `/login.html` |

### Admin Auth

Admin sessions are stored separately under `zm_admin_session`. Brute-force protection locks the account for 15 minutes after 5 failed attempts — lockout state is stored in `sessionStorage` (per-tab, not clearable via Application tab tricks).

| Function | Description |
|----------|-------------|
| `adminLogin(email, pw)` | Validates credentials (PBKDF2); sets session |
| `requireAdmin()` | Redirects to login if not authenticated |
| `getAdminSession()` | Returns admin session or `null` |
| `adminLogout()` | Clears admin session and redirects |

### Cart State

Cart items stored in `zm_cart`. Each item:

```js
{
  key:       'PRD-0001__Red',   // productId + '__' + variant
  productId: 'PRD-0001',
  name:      'Product Name',
  slug:      'product-slug',
  price:     2990,
  image:     'https://...',
  variant:   'Red',
  qty:       2,
  stock:     50,
}
```

### Cart Functions

| Function | Description |
|----------|-------------|
| `getCart()` | Returns current cart array |
| `getCartCount()` | Total item quantity |
| `getCartTotal()` | Total price (no shipping) |
| `addToCart(product, qty, variant)` | Adds or increments item |
| `removeFromCart(key)` | Removes item |
| `updateQty(key, qty)` | Sets quantity (removes if qty < 1) |
| `clearCart()` | Empties the cart |

### Coupon Types

| Type | Effect |
|------|--------|
| `percent` | Reduces subtotal by X% |
| `fixed` | Reduces subtotal by fixed Rs. amount |
| `shipping` | Sets shipping cost to Rs. 0 |

Coupons support a `minOrder` threshold and a `usageLimit`. Usage count is incremented and the coupon deactivated when the limit is reached.

## 9. Product Reviews System

### Rules

- Customer must be **logged in** (registered account)
- Customer must have a **paid / processing / shipped / delivered order** containing that product
- **One review per user per product**
- Reviews can be **edited exactly once** (editing resets to pending)
- All reviews start as `approved: false` — admin must approve before public display

### Review Lifecycle

```
Customer submits → approved: false (pending)
       ↓
Admin → Reviews → Approve → approved: true → visible on product page
       ↓
Admin → Reviews → Reject  → rejected: true → hidden everywhere
```

### Review Functions

| Function | Description |
|----------|-------------|
| `addReview({ productId, userId, ... })` | Submits new review (starts pending) |
| `editReview({ productId, userId, ... })` | One-time edit (resets to pending) |
| `approveReview(productId, reviewId)` | Admin approve |
| `rejectReview(productId, reviewId)` | Admin reject |
| `getProductReviews(productId)` | Returns approved reviews for a product |
| `getAllReviews()` | Returns map of all reviews keyed by productId |
| `hasPurchased(userId, productId)` | Checks if user has a valid order |
| `getReviewStats(productId)` | Returns `{ avg, count, breakdown }` |

## 10. JavaScript Architecture

### `withLoader()` Pattern

Every page wraps its init in `withLoader()`:

```js
import { withLoader } from './loader.js';
import { injectLayout } from './layout.js';

withLoader(async () => {
  injectLayout({});
  // page logic here
});
```

`withLoader` shows the spinner, runs the async function, hides the spinner. Safety nets:
- Loader always dismisses after 1.5 seconds maximum even if an error is thrown
- `window.addEventListener('error', hideLoader)` and `unhandledrejection` listeners
- DOMContentLoaded fallback: if JS never calls `withLoader`, loader hides after 800ms

### Admin Page Pattern

```js
withLoader(async () => {
  if (!requireAdmin()) return;  // redirects to login if not authed
  injectAdminLayout('Page Title');
  // admin page logic
});
```

### Store Functions (`js/store.js`)

| Function | Returns | Description |
|----------|---------|-------------|
| `getProducts()` | `Product[]` | Merges default + edited + added products |
| `saveProduct(p)` | — | Saves/updates a product |
| `deleteProduct(id)` | — | Removes from all product stores |
| `getOrders()` | `Order[]` | All orders (demo seed if empty) |
| `saveOrders(arr)` | — | Replaces the orders array |
| `getUsers()` | `User[]` | All users (demo seed if empty) |
| `saveUsers(arr)` | — | Replaces the users array |
| `getCategories()` | `Category[]` | All categories |
| `getCoupons()` | `Coupon[]` | All coupons |
| `getShippingZones()` | `Zone[]` | Editable shipping zones |
| `getShippingRate(district)` | `number` | Delivery cost in Rs. |
| `getDeliveryDays(district)` | `string` | e.g. `"1–2"` |

### Security Utilities (`js/security-utils.js`)

| Export | Purpose |
|--------|---------|
| `esc(str)` | HTML-escape for `innerHTML` insertion |
| `escAttr(str)` | HTML-escape for attribute values |
| `escJs(str)` | Escape for inline JS string context |
| `createEl(tag, props)` | Safe DOM element builder (no innerHTML) |
| `html(strings, ...values)` | Tagged template — auto-escapes all interpolated values |
| `safeUrl(url, fallback)` | Blocks `javascript:` and protocol-relative URLs |
| `safeRedirectPath(raw)` | Validates `?next=` redirect target |
| `sanitizeInput(str, max)` | Strips null bytes and control characters |
| `hashPassword(plain)` | PBKDF2 (310,000 iterations) hash via Web Crypto API |
| `verifyPassword(plain, stored)` | Compares plain text against stored hash |
| `checkBruteForce()` | Returns error string if locked out, else `null` |
| `recordFailedAttempt()` | Increments failure counter |
| `clearFailedAttempts()` | Resets counter on successful login |

4. Create a backend function (Supabase Edge Function or server-side route) to generate the MD5 payment hash using the merchant secret — **never generate this hash in the browser**
5. Create a webhook handler for the PayHere notify URL to confirm payments server-to-server
6. Replace the demo simulation in `checkout.js` with real PayHere SDK calls:

```js
// Add to checkout.html <head>:
// <script src="https://www.payhere.lk/lib/payhere.js"></script>

payhere.startPayment({
  sandbox:     false,
  merchant_id: PAYHERE_MERCHANT_ID,
  return_url:  window.location.origin + '/order-success.html',
  cancel_url:  window.location.origin + '/checkout.html',
  notify_url:  'https://YOUR_PROJECT.supabase.co/functions/v1/payhere-notify',
  order_id:    order.id,
  items:       order.items.map(i => i.name).join(', '),
  amount:      order.total.toFixed(2),
  currency:    'LKR',
  hash:        await getPaymentHash(order),  // from your backend
  first_name:  formData.firstName,
  last_name:   formData.lastName,
  email:       user.email,
  phone:       formData.phone,
  address:     formData.addressLine1,
  city:        formData.city,
  country:     'Sri Lanka',
});
```

## 17. Security

### Vulnerabilities Fixed (v29)

| ID | Vulnerability | Fix |
|----|--------------|-----|
| CRIT-01 | Admin credentials compared in plain text | PBKDF2 hashing (310k iterations) |
| CRIT-02 | Passwords stored in plain text | PBKDF2 hash on registration; auto-migration on legacy login |
| CRIT-03 | Session fixation | Fresh `crypto.randomUUID()` token on every `setSession()` |
| CRIT-04 | XSS in admin order detail | All user fields escaped with `esc()` |
| CRIT-05 | XSS in admin users table | All user fields escaped with `esc()` |
| CRIT-06 | XSS in search dropdown | Query string and product slugs escaped |
| CRIT-07 | Price manipulation via localStorage cart | Prices re-fetched from canonical product catalog at submission |
| CRIT-08 | Coupon discount bypassed via sessionStorage | Coupon re-validated from `getCoupons()` at submission |
| CRIT-09 | Open redirect via `?next=` parameter | `safeRedirectPath()` rejects absolute and protocol-relative URLs |
| CRIT-10 | Bank slip localStorage quota bomb | Max 500 KB; SVG blocked; MIME + extension validation |
| HIGH-01 | XSS via blog post content | DOMPurify sanitisation with strict allowlist |
| HIGH-02 | No Content Security Policy | Full CSP header in `vercel.json` |
| HIGH-05 | No brute-force protection | 5 failures → 15-minute lockout; server-side state |
| HIGH-06 | `javascript:` protocol in social links | All hrefs validated with `safeUrl()` |
| MED-03 | XSS in cart rendering | All cart fields escaped before `innerHTML` |

### Security Headers (`vercel.json`)

- `Strict-Transport-Security` — HSTS (2 years, includeSubDomains, preload)
- `X-Frame-Options: DENY`
- `X-Content-Type-Options: nosniff`
- `X-XSS-Protection: 1; mode=block`
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Permissions-Policy` — camera, microphone, geolocation all blocked
- `Content-Security-Policy` — restricts script, style, image, connect, font, and frame sources
- `Cross-Origin-Opener-Policy: same-origin-allow-popups`
- Admin pages: `X-Robots-Tag: noindex, nofollow, noarchive` + `Cache-Control: no-store`

### Remaining Limitations (Require Architecture Changes)

| Issue | Requires |
|-------|---------|
| Admin auth is entirely client-side | Supabase Row Level Security |
| PayHere integration is a simulation | Real PayHere SDK + server-side hash |
| All data in localStorage | `DEMO_MODE = false` + Supabase |
| No build tool (no minification) | Vite or similar bundler |

```sql
ALTER TABLE products         ENABLE ROW LEVEL SECURITY;
ALTER TABLE orders           ENABLE ROW LEVEL SECURITY;

ALTER TABLE coupons          ENABLE ROW LEVEL SECURITY;

ALTER TABLE shipping_zones   ENABLE ROW LEVEL SECURITY;

-- Public reads
CREATE POLICY "Public read active products"  ON products   FOR SELECT USING (active = true);
CREATE POLICY "Public read categories"       ON categories FOR SELECT USING (true);
CREATE POLICY "Public read shipping zones"   ON shipping_zones FOR SELECT USING (true);
CREATE POLICY "Public read approved reviews" ON reviews    FOR SELECT USING (approved = true);

-- Authenticated customer access
CREATE POLICY "Users insert own orders"  ON orders   FOR INSERT WITH CHECK (customer_id = auth.uid()::text);
CREATE POLICY "Users view own orders"    ON orders   FOR SELECT USING (customer_id = auth.uid()::text);
CREATE POLICY "Users view own profile"   ON profiles FOR SELECT USING (id = auth.uid());
CREATE POLICY "Users update own profile" ON profiles FOR UPDATE USING (id = auth.uid());
CREATE POLICY "Users submit reviews"     ON reviews  FOR INSERT WITH CHECK (user_id = auth.uid()::text);
```

## 22. Troubleshooting

### Black Screen / Loader Never Dismisses

**Causes and fixes:**

1. **Opening via `file://`** — ES Modules require HTTP. Use a local server (VS Code Live Server, `npx serve .`, or `python3 -m http.server`).
2. **JavaScript module error** — open DevTools (F12) → Console tab. Look for red errors.
3. **Admin session expired** — go to `/admin/index.html` and log in again.

The safety net in `loader.js` forces the loader away after 1.5 seconds even if a module crashes. If the screen clears after a moment but content is missing, check the Console for the underlying error.

### Dashboard Chart Area Black / Sections Stuck on "Loading…"

This was a known bug fixed in v29. If you see it after updating:

1. Check `css/pages/admin-dashboard.css` — confirm there is **no** `content-visibility: auto` rule on `.dash-section`. This CSS property caused chart canvases to report `width: 0`.
2. Check `js/admin/admin-dashboard.js` — confirm the chart poll deadline is `3000` ms (not `500` ms).
3. Hard-refresh the browser (`Ctrl+Shift+R`) to clear any cached old JS files.

### New Registrations Not Appearing in Admin

Open DevTools → Application → Local Storage. Confirm `zm_users` exists and contains the registered user. If it doesn't, you may be using an older version of `auth.js`. The current version syncs new accounts to both `zm_registered_users` and `zm_users`.

### Shipping Rate Changes Not Taking Effect

Confirm you are using the current `store.js`. Earlier versions read directly from the hardcoded `SHIPPING_ZONES` constant. The current version calls `getShippingZones()` which reads `zm_shipping_zones` first.

### Products Not Showing in Shop

1. Admin → Products — confirm the product has **Active** ticked
2. Confirm price > 0
3. Confirm at least one image URL is set

### WhatsApp Message Not Opening

1. Confirm `WA_PHONE` in `config.js` is digits only (no `+`, no spaces, no dashes)
2. Check your browser is not blocking pop-ups for localhost
3. Allow pop-ups for your domain in browser settings

### Admin Password Forgotten

```js
// Run in browser DevTools console:
localStorage.removeItem('zm_admin_password_hash');
location.reload();
// Password is now back to 'admin123'
```

### Reset All Demo Data

```js
// Run in browser DevTools console:
Object.keys(localStorage)
  .filter(k => k.startsWith('zm_'))
  .forEach(k => localStorage.removeItem(k));
location.reload();
```

### CORS / ES Module Errors

You are opening the HTML file directly via `file://`. Use a local server as described in the Quick Start section.

### Cart Summary Crashes on Some Pages

Ensure you are using the current `cart.js`. Older versions called `.querySelector()` on a `null` element when a page had no cart summary wrapper. The current version uses `getElementById()` with null-guards.

### 25a. Vercel Serverless Functions

All API routes live in `/api/` and are auto-deployed by Vercel.

**Security model:**
- Admin routes (`/api/admin/*`) require `X-Admin-Token` header matching `ADMIN_API_TOKEN` env var
- All admin routes use `SUPABASE_SERVICE_ROLE_KEY` (bypasses RLS) — never exposed to browser
- PayHere webhook verifies MD5 signature using `PAYHERE_MERCHANT_SECRET`
- CORS restricted to `SITE_URL` env var

**New env vars required (Vercel → Settings → Environment Variables):**

```
SUPABASE_SERVICE_ROLE_KEY   # Project Settings → API → service_role
ADMIN_API_TOKEN             # openssl rand -hex 32
PAYHERE_MERCHANT_SECRET     # PayHere → Settings → Merchant Secret
SITE_URL                    # https://yourdomain.com
```

**PayHere IPN setup:**
1. Log in to PayHere → Settings → Domains & URLs
2. Set **Notify URL** to: `https://yourdomain.com/api/payhere-webhook`
3. Add `PAYHERE_MERCHANT_SECRET` to Vercel env vars

## 26. Bug Fixes — v29.1 (April 2026)

> These fixes were applied after the v29 merge of the two codebases.

### Bug 1 — Coupon Usage Tracking Never Incremented (`js/checkout.js`)

**Symptom:** Coupons could be used an unlimited number of times regardless of their `maxUses` limit. Usage limits set in Admin → Coupons had no effect.

**Root cause:** The coupon schema defines `used` and `maxUses` fields, but checkout was reading and writing `usageCount` and `usageLimit` (undefined properties). Every order silently incremented `undefined + 1 = 1` into a phantom property that was never stored.

**Fix:** Updated checkout coupon tracking to use the correct schema fields:
```js
// Before (broken):
coupon.usageCount = (coupon.usageCount || 0) + 1;
if (coupon.usageLimit && coupon.usageCount >= coupon.usageLimit) coupon.active = false;

// After (fixed):
coupon.used = (coupon.used || 0) + 1;
if (coupon.maxUses && coupon.used >= coupon.maxUses) coupon.active = false;
```

---

### Bug 2 — Free-Shipping Coupons Never Applied at Checkout (`js/checkout.js`)

**Symptom:** The `FREESHIP` coupon type (`type: 'shipping'`) displayed "Free shipping" correctly on the cart page, but at checkout the shipping cost was never zeroed — customers were always charged the full shipping rate even with a valid shipping coupon.

**Root cause:** The checkout computed `freeShip = !!coupon.freeShipping`. The `freeShipping` property does not exist on the coupon schema — only `type` does. So `!!undefined` is always `false`.

**Fix:**
```js
// Before (broken):
freeShip = !!coupon.freeShipping;  // always false

// After (fixed):
freeShip = coupon.type === 'shipping';  // correct
```

---

### Bug 3 — DEFAULT_ORDERS Contained Non-Existent Products (`js/store.js`)

**Symptom:** Demo users (Dinusha, Kasun, Shalini) could not write reviews for certain products because `hasPurchased()` checked their order history against real catalog IDs — but the orders referenced products like `PRD-0012 "Stainless Steel Water Bottle"`, `PRD-0010 "Yoga Mat"`, `PRD-0011 "Protein Shaker"`, and `PRD-0019 "Samsung 4K Smart TV"` which don't exist in `DEFAULT_PRODUCTS`.

**Root cause:** The extra delivered orders (ORD-20240004 through ORD-20240009) were copied from an older catalog version that had different products.

**Fix:** Replaced all 6 extra orders with items that map to actual catalog products (PRD-0001 through PRD-0020). Corrected `subtotal`, `shipping`, and `total` values to match the real product prices.
